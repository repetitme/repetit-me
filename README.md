# RepetitMe

**Описание работы приложения**: В приложении RepetitMe ученики смогут найти себе репетитора по конкретному учебному предмету, записаться на занятие, написать поделиться впечатлениями о прошедшем уроке и о самом репетиторе. Репетиторы, в свою очередь, создают подробный профиль, с заполнением информации о себе и о своем образовании, подтверждая его документом. Они могут принимать заявки от учеников и запланировать урок на любое время.

**Стек технологий**: React, Redux, TypeScript, SCSS, Vite, HTML, ESLint, Prettier, Stylelint.

## Подготовка к работе:

Для начала работы в приложении или запуска, необходимо выполнить следующие указания:

1. Клонируем репозиторий

```
git clone git@github.com:repetitme/repetit-me.git
```

2. Переходим в ветку для разработки (develop)

```
git checkout develop
```

3. Устанавливаем зависимости

```
yarn
yarn dev
```

4. Сборка проекта (при необходимости)

```
yarn build
```

## Запуск в Docker:

1. Сборка Docker-образа

```
yarn docker:build
```

2. Запуск контейнера

```
yarn docker:run
```

3. Переходим по адресу http://localhost:8080

### Запуск в Docker на сервере

- Запуск Docker-compose

```
yarn compose:up
```

### Необязательные действия

- Остановить контейнер

```
docker stop repetit-me
```

или

```
ctrl+c в консоли
```

- Удалить контейнер

```
docker rm repetit-me
```

- Удалить образ

```
docker rmi repetit-me-frontend
```

- Очистка неиспользуемых образов и ресурсов для освобождения места

```
docker builder prune
```

## Правила работы над проектом:

1. Без согласования и код-ревью нельзя пушить в develop и main.
2. Все изменения вносятся через PR, который обязательно проходит код-ревью.
3. Все PR должны быть открыты в develop, кроме случаев когда нужно срочно в main.
4. Все PR должны быть открыты из ветки, которая была создана из develop. Перед PR в эту ветку должны быть слиты или ребазированы последние изменения develop.
5. Создавайте PR только после того как закончили работу над фичей или фиксом. Для этого проверьте что код собирается в dist и не ломает работу других частей проекта. Обязательно проверяйте линтером и применяйте форматирование.
6. Время работы над задачей не более трех дней. Если задача не закончена за это время, то нужно открыть PR с текущим результатом и попросить помощи у тимлида или в чате проекта, чтобы скорректировать задачу, поделить ее на более мелкие, помочь с решением или передать кому-то другому.
7. Строго не должно быть изменений в yarn файлах (.yarn, yarn.lock).
8. Установка зависимостей через yarn.

### Правила формирования коммитов:

Во время работы не делайте больших коммитов, создавайте маленькие атомарные коммиты с фиксированным функционалом, описывайте что сделали в комментарии к коммиту. Правильная практика - добавил новый компонент (_название компонента_), исправил верстку в шапке. Нежелательная - "Исправил ошибки", "Добавил компоненты", "Рефакторинг кода". Хорошо было бы разбивать каждую фичу на несколько маленьких коммитов.

При именовании коммитов используйте следующие конвенционные префиксы:

- `feat:` добавление нового функционала
- `fix:` исправление бага
- `refactor:` рефакторинг кода
- `chore:` изменения не связанные с продакшен кодом, например тесты, документация, настройка сборки

**Пример**: `feat: добавление нового компонента Card`, `fix: исправление бага в компоненте Card`, `refactor: рефакторинг компонента Card`, `chore: задокументирован компонент Card`

### Правила оформления Pull Request (PR):

Как должна выглядеть структура PR:

- именование веток: {тип_задачи}/{краткое-описание-задачи}:
  - тип_задачи: feat, fix, refactor, chore;
  - краткое-описание-задачи: например, "Добавление нового компонента Card";
- размер ПР: делать небольшие итерации, большие ПР трудно смотреть, долго вникать в суть задачи;
- описание ПР (техническое описание), скрин с рабтающей фичей;

#### Существуют следующие приоритеты для задач:

- P-LOW - можно посмотреть пр в течении 2-3 дней.
- P-HIGH - необходимо посмотреть в течении 1-2 дней.
- P-ASAP - срочные изменения или фикс, который решает проблему на бою, не
  требуют отлагательств.

#### Существуют следующие размеры задач:

- S-SMALL - небольшая доработка, содержащая изменения от пары строк, до
  изменений в одном или нескольких модулях
- S-MEDIUM - доработка, содержащая изменения в нескольких модулях
- S-BIG - большая доработка, содержащая изменения в нескольких модулях или
  большую доработку

**Пример:** `[P-HIGH] [S-BIG] : сделал кнопку`

## Структура проекта:

- `src:` Корневая директория проекта, в которой содержится основная логика приложения.
- `src/app:` Директория, в которой расположены главный файлы приложения для входа в него, а так же его состояние.
- `src/assets:` Директория, в которой расположены статические файлы, такие как картинки, стили, шрифты и другие ресурсы.
- `src/config:` Директория, в которой расположены конфигурационные файлы для приложения.
- `src/entities:` Директория, в которой расположены сущности приложения, такие как пользователи, уроки и другие объекты.
- `src/features:` Директория, в которой расположены переиспользуемые функции для работы приложения.
- `src/pages:` Директория, в которой расположены компоненты страниц приложения.
- `src/shared:` Директория, в которой расположены переиспользуемые компоненты, хуки, утилиты и типы для приложения.
- `src/widgets:` Директория, в которой расположены самостоятельные компоненты, которые можно использовать в любой точке приложения.

## Архитектура проекта:

Приложение построено на архитектуре **FSD (Feature-Sliced Design)**, что упрощает организацию кода и его поддержку. Этот подход позволяет разделить приложение на независимые функциональные модули, что способствует лучшей управляемости и масштабируемости. [Подробнее об архитектуре](https://feature-sliced.design/docs).
